# HACKINTOS

Overview
--------

HACKINTOS is a lightweight cyber-terminal web application designed to provide a developer-focused browsing and inspection experience inside a single web app. It combines an HTML viewer, a configurable HTTPS client, network and request/response inspection tools, a console, and an integrated terminal, all presented in a dark cyberpunk-inspired UI. The core architecture uses a Node.js and Express backend to perform network requests and serve assets, while the frontend is built with plain HTML, CSS, and JavaScript and is designed to be modular and plugin-ready.

This README is a comprehensive reference for developers who want to run, extend, package, and distribute HACKINTOS. It includes configuration details, build and distribution instructions, security considerations, API references, and developer workflow guidance.

Why HACKINTOS?
--------------

Developers and security researchers frequently need a compact, self-hosted toolset that lets them fetch and inspect web pages, analyze headers and responses, run arbitrary HTTP requests, and interact with a page's storage and cookies. HACKINTOS provides these features in one place without relying on full browser automation tooling. It is useful for rapid debugging, testing, learning HTTP, and building offline demonstrations.

Key Features
------------

- HTML Viewer: Render HTML responses in an isolated sandboxed iframe for inspection.
- HTTPS Client: Send GET/POST/PUT/DELETE requests via the server proxy (avoids CORS limitations).
- Network Inspector: View timing, status codes, headers, and raw response bodies.
- Console: Aggregate client-side and server-side messages in a single console panel.
- Storage Inspector: View LocalStorage / SessionStorage and Cookies for the loaded page.
- Terminal Interface: Lightweight terminal integration using `xterm.js` (pluggable).
- Plugin Ready: Modular frontend and server structure to enable future plugins.
- Packaging: Scripts to produce a deployable `dist/` and options to bundle as an executable.

Project Structure
-----------------

Top-level layout (key folders):

- `client/` — Frontend HTML, CSS, JS and assets.
- `server/` — Express server, controllers, and backend utilities.
- `public/` — Static public assets for runtime.
- `scripts/` — Helper scripts (e.g., `build.js` to produce `dist/`).
- `dist/` — Generated by the build script for distribution.

Getting Started (Development)
-----------------------------

Prerequisites

- Node.js (v18+ recommended)
- npm (or yarn)

Install dependencies:

```bash
cd D:/Projects/HACKINTOS
npm install
```

Run in development mode (auto-reloads with `nodemon`):

```bash
npm run dev
```

Open the app at:

```
http://localhost:3000
```

Configuration (.env)
---------------------

Copy `.env.example` to `.env` and update values for your environment. The `.env` file is read by the server at runtime. Example variables and purpose:

- `PORT` — Port the Express server listens on (default 3000).
- `NODE_ENV` — `development` or `production`, controls logging and debug behaviors.
- `SESSION_SECRET` — A long random string used if session middleware is enabled.
- `PROXY_TIMEOUT` — Milliseconds to wait for proxied HTTP requests.
- `ALLOW_SELF_SIGNED` — Allow self-signed HTTPS responses (use carefully).

Security note: Do not commit `.env` to version control. Use `.env.example` as a template and keep secrets out of git.

How the Proxy Works
-------------------

The backend exposes a simple endpoint at `/api/http-proxy` that accepts a JSON payload specifying `method`, `url`, `headers`, and `data`. The server performs the specified HTTP request using `axios` and returns a sanitized response with `status`, `headers`, and `data`. This approach bypasses CORS restrictions and centralizes network requests in a controlled server environment.

Example request payload (POST to `/api/http-proxy`):

```json
{
	"method": "GET",
	"url": "https://example.com",
	"headers": {},
	"data": null
}
```

Packaging and Distribution
-------------------------

A basic build script is provided in `scripts/build.js`. Running `npm run dist` will copy `server/`, `client/`, `public/`, and `README.md` into a `dist/` folder and write a minimal `package.json` there ready for production installation. Typical steps to create a runnable distribution:

```bash
npm run dist
cd dist
npm install --production
npm start
```

Creating a single-file Windows executable
---------------------------------------

If you want a single `.exe` file that runs the server, there are two common approaches:

1. Use `pkg` or `nexe` to bundle Node.js and your code into one executable. This is fast and convenient but can be tricky with dynamic file reads or native modules. You may need to include `assets` and adjust the `pkg` configuration. Example:

```bash
npm install --save-dev pkg
npx pkg . --targets node18-win-x64 --output dist/hackintos.exe
```

2. Use Electron if you prefer a desktop UI (creates a GUI app rather than a server). Electron provides a more native-feeling application and is ideal if you want a standalone desktop browser/inspector experience.

Be aware of tradeoffs: `pkg` bundles everything into a single binary but can make updating assets harder, whereas `dist/` with `npm install` keeps the node_modules separate and simpler to update.

Runtime and Deployment
----------------------

For production, use a process manager (PM2, systemd, or a container platform) to ensure the service stays up and can be restarted on failure. Example with `pm2`:

```bash
pm2 start server/app.js --name hackintos --env production
```

If deploying behind a public-facing web server (NGINX, Apache), configure reverse proxying to the `PORT` used by HACKINTOS and enable TLS termination there. Keep the Node process in a private network or behind firewall rules where appropriate.

Filesystem and Assets
---------------------

The server serves static files from `client/` and `public/`. When packaging with `pkg` or similar, ensure you include the `client/**/*` and `public/**/*` patterns in the bundler configuration (see `package.json` `pkg.assets` entry). If you keep a separate `dist/` folder, copying assets is sufficient and simpler to manage.

Extending HACKINTOS (Plugins)
----------------------------

HACKINTOS is designed so additional features can be added without modifying core files extensively. Recommended extension points:

- Frontend plugins: place modular scripts under `client/components/` and load them via a plugin registry or manifest.
- Backend services: add new routes under `server/routes/` and controllers under `server/controllers/`.
- WebSocket events: extend `io.on('connection')` handlers to add real-time events for plugins.

When designing a plugin, keep the API surface minimal and use namespaced event channels or REST routes to avoid collisions with core functionality.

API Reference (Key Endpoints)
-----------------------------

- `GET /` — Serves the main client UI (index.html).
- `POST /api/http-proxy` — Proxies HTTP requests. Request body: `{ method, url, headers, data }`. Response includes `status`, `statusText`, `headers`, `data`.

Socket.IO
---------

Socket.IO is used for live status updates and future real-time features (console streaming, network updates). Currently the server emits a `status` event on new connections with a small payload. Clients can extend this to receive live logs or plugin messages.

Frontend Architecture
---------------------

The frontend uses plain HTML templates in `client/` and small JavaScript modules under `client/assets/js/`. Key modules:

- `httpClient.js` — Wraps calls to `/api/http-proxy`.
- `main.js` — Handles UI events, navigation, and panel management.

Planned enhancements include integrating Monaco Editor for source viewing and `xterm.js` for an improved terminal interface; both are already referenced as dependencies to be added.

Development Workflow
--------------------

1. Branch from `main` for new features.
2. Create a concise feature branch name and include small, atomic commits.
3. Add unit tests where practical (server controllers and utilities).
4. Open a pull request with a short description and testing steps.

Testing
-------

This initial scaffold doesn't include a test harness; consider adding Jest or Mocha for unit tests. Suggested test targets:

- `server/controllers/httpController.js` — mock axios and validate responses for various status codes and timeouts.
- `client/assets/js/*` — use a headless browser or JSDOM to validate UI logic.

Troubleshooting
---------------

- If the proxy returns an error, check `PROXY_TIMEOUT` and network rules.
- If `pkg` or other bundlers fail, inspect `package.json` `pkg` entries and ensure referenced assets exist.
- For CORS-like issues, ensure all client-side network calls use the backend proxy rather than direct cross-origin requests.

Security Considerations
-----------------------

HACKINTOS allows arbitrary requests to external URLs. Running it in an untrusted environment could be a security risk. Recommendations:

- Run on a trusted local network and avoid exposing it publicly without authentication and robust request filtering.
- Validate and sanitize user-provided URLs in the backend if exposing to multiple users.
- Consider rate-limiting and request logging to detect misuse.
